
📘 Техническое задание

Проект: Методология разработки IT-продуктов с использованием AI

⸻

1. Цель разработки

Создать методологию, которая позволяет:

из идеи или сырого ТЗ получать полное, стандартизированное техническое задание и комплект проектной документации, достаточный для разработки полноценного продукта (не MVP)

⸻

2. Область применения

Методология должна применяться для:
	•	веб-приложений
	•	backend/API сервисов
	•	AI-систем
	•	интеграционных решений

📌 Ограничение:
	•	не привязана к конкретному стеку технологий
	•	не зависит от команды (может использоваться индивидуально или в команде)

⸻

3. Входные данные

Методология должна работать с любым уровнем входа:

3.1 Типы входа:
	1.	💡 Идея (1–2 предложения)
	2.	📝 Сырой текст (хаотичное описание)
	3.	📄 Частичное ТЗ
	4.	📚 Готовое ТЗ (для нормализации и проверки)

⸻

4. Выходные данные (ключевое требование)

Методология должна приводить к формированию полного набора артефактов:

⸻

📦 4.1 Основной пакет документации

1. Product Definition Document (PDD)
	•	описание проблемы
	•	целевая аудитория
	•	пользовательские сценарии
	•	метрики успеха

⸻

2. Functional Specification (FS)
	•	функциональные требования
	•	бизнес-логика
	•	сценарии работы системы

⸻

3. System Design Document (SDD)
	•	архитектура системы
	•	компоненты
	•	взаимодействия
	•	модель данных

⸻

4. Non-Functional Requirements (NFR)
	•	производительность
	•	безопасность
	•	масштабируемость
	•	отказоустойчивость

⸻

5. API Specification
	•	эндпоинты
	•	контракты
	•	форматы данных

⸻

6. Data Model
	•	сущности
	•	связи
	•	ограничения

⸻

7. UX / User Flows
	•	пользовательские сценарии
	•	логика интерфейса

⸻

8. Execution Plan
	•	декомпозиция задач
	•	зависимости
	•	этапы реализации

⸻

9. Test Strategy
	•	виды тестирования
	•	ключевые сценарии
	•	критерии качества

⸻

10. Validation Criteria
	•	как определить, что продукт “готов”

⸻

5. Требования к методологии

⸻

5.1 Структурные требования

Методология должна:
	•	быть этапной
	•	иметь чёткую последовательность действий
	•	включать контрольные точки (gates)
	•	обеспечивать проверяемость результатов

⸻

5.2 Логические требования

Методология должна:
	•	устранять неоднозначности
	•	выявлять противоречия
	•	дополнять недостающую информацию
	•	нормализовать входные данные

⸻

5.3 Требования к качеству

Результат методологии должен:
	•	быть однозначно интерпретируем
	•	быть достаточным для разработки без дополнительных уточнений
	•	исключать “догадки” разработчика
	•	обеспечивать предсказуемый результат

⸻

5.4 Требования к AI-интеграции

Методология должна:
	•	предусматривать использование AI на каждом этапе
	•	определять:
	•	где AI допустим
	•	где требуется проверка
	•	где решения принимает человек

⸻

6. Этапы методологии (черновая структура)

⸻

Этап 0 — Normalization

👉 приведение входных данных к структурированному виду

⸻

Этап 1 — Problem Formalization

👉 формализация проблемы и целей

⸻

Этап 2 — Requirements Extraction

👉 выделение требований

⸻

Этап 3 — System Modeling

👉 моделирование логики системы

⸻

Этап 4 — Architecture Design

👉 проектирование архитектуры

⸻

Этап 5 — Specification

👉 формирование полной документации

⸻

Этап 6 — Planning

👉 декомпозиция и план реализации

⸻

Этап 7 — Validation

👉 проверка полноты и качества

⸻

7. Контрольные точки (обязательное требование)

Каждый этап должен иметь:
	•	критерии входа
	•	критерии выхода
	•	чек-лист проверки

⸻

8. Ограничения

Методология НЕ должна:
	•	зависеть от конкретных инструментов
	•	требовать конкретного языка программирования
	•	включать реализацию (код)

⸻

9. Критерии успешности методологии

Методология считается успешной, если:
	1.	Из идеи можно получить полный комплект документации
	2.	Разработчик может реализовать продукт без дополнительных вопросов
	3.	Разные исполнители приходят к схожему результату
	4.	Снижается количество ошибок на этапе разработки

⸻

10. Дополнительные требования (важно)

Методология должна быть:
	•	расширяемой
	•	пригодной для автоматизации (в виде AI-агентов / skills)
	•	формализуемой (чтобы можно было превратить в систему)
